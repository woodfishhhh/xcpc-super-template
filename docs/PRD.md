# 超级板子 PRD

## 1. 产品定位

“超级板子”是一个离线优先的 ACM/XCPC 打印稿生成工作台。

它不是新的 OI Wiki、博客系统或算法百科。它的核心任务是：让选手从公共板子库和自己的本地板子库中快速勾选内容，编辑、排序、配置版式，然后生成可打印的 Markdown/PDF。

第一版原型只定义和验证工作流，不做大规模内容搬运。

## 2. 原型目标

第一版原型需要证明这件事成立：没有账号、没有后端，也能完成一份比赛打印稿的完整装配流程。

核心流程：

1. 浏览公共模板库。
2. 勾选需要的模板。
3. 新增或编辑个人模板。
4. 调整模板顺序。
5. 选择目录、介绍详细度、版式和导出格式。
6. 预览生成结果。
7. 下载 Markdown 和/或 PDF。
8. 断网后仍可打开应用，继续导入、编辑、生成。

第一屏必须是可用的工作台，不做营销首页。

## 3. 目标用户

- 准备现场赛打印稿的 ACM/XCPC 选手。
- 想共享一套公开基础板子，但又保留个人/队伍模板的队伍。
- 不想登录、不想云同步、希望数据在自己电脑上的用户。

## 4. 非目标

- 不做账号系统。
- 不做云端同步。
- 不做评论、点赞、用户主页或社区功能。
- 不接 Online Judge。
- 不做完整 OI Wiki 风格的教学文章系统。
- 第一版不要求完整迁移外部仓库中的所有算法模板。

## 5. 仓库与内容组织

公共模板库与应用代码放在同一个仓库中。

公共模板库目录命名为 `板子/`。目录结构按照算法分类自然展开，例如：

```text
板子/
  图论/
    二分图/
      meta.json
      code.md
    强连通分量/
      meta.json
      code.md
  字符串/
    KMP/
      meta.json
      code.md
```

最底层的每个算法文件夹包含两个核心文件：

- `meta.json`：存放标题、分类、顺序、复杂度、标签、简介、来源、许可证等元数据。
- `code.md`：存放该模板的代码正文，使用 Markdown 代码块，便于直接生成文档。

这样做的目的：

- 文件结构和人的理解一致，打开仓库就能看懂。
- 每个算法模板可以独立修改、审查、溯源。
- 后续可以从 `meta.json + code.md` 自动生成应用内索引。
- GitHub 上直接浏览也比较舒服。

## 6. 模板元数据

`meta.json` 建议结构：

```json
{
  "id": "graph.bipartite.matching",
  "title": "二分图最大匹配",
  "category": ["图论", "二分图"],
  "learningOrder": 210,
  "tags": ["graph", "bipartite", "matching"],
  "timeComplexity": "O(nm)",
  "spaceComplexity": "O(n + m)",
  "brief": "用于在二分图中寻找最大匹配。",
  "detail": "常用于配对、分配、最小点覆盖等问题。打印稿中保留建图方式、调用入口和结果解释。",
  "codeLanguage": "cpp",
  "source": {
    "name": "Original or reference source",
    "url": "https://example.com",
    "license": "Unknown"
  },
  "updatedAt": "2026-06-10"
}
```

`code.md` 示例：

````markdown
```cpp
// code here
```
````

## 7. 公共模板库

默认顶级分类参考 OI/XCPC 常见组织方式：

- 基础算法。
- 数据结构。
- 图论。
- 字符串。
- 数学与数论。
- 几何。
- 多项式。
- 动态规划。
- 网络流。
- 杂项。

默认排序使用学习顺序，从易到难。用户也可以选择字典序或手动顺序。

## 8. 选择与排序

用户可以从公共模板库和个人模板库中选择模板。

排序方式：

- 默认：学习顺序。
- 可选：字典序。
- 可选：当前打印稿内手动排序。

手动排序同时支持：

- 鼠标拖拽。
- 键盘移动按钮，例如上移、下移、移到顶部、移到底部。

拖拽可以使用现成 Vue 生态库实现；键盘移动按钮作为必须项保留，避免只有拖拽导致键盘用户无法操作。

## 9. 介绍详细度

每个选中的模板可以选择：

- 不显示介绍。
- 显示简略介绍。
- 显示详细介绍。

介绍默认放在代码前面。

## 10. 个人模板库

个人模板库第一版就要支持文件夹/分类。

用户可以：

- 新增个人模板。
- 编辑个人模板。
- 删除个人模板。
- 给个人模板设置分类路径。
- 把个人模板插入当前打印稿。
- 将个人模板库导出为 JSON。
- 从 JSON 导入个人模板库。

个人模板保存在浏览器本地。建议使用 IndexedDB，而不是只用 localStorage。

导入冲突策略第一版可以简单处理：如果导入模板的 ID 已存在，则自动生成新的本地 ID，并保留原标题和分类。

## 11. 打印稿配置

用户可以配置：

- 打印稿标题。
- 是否生成目录。
- 目录深度，至少支持 1 级和 2 级。
- 输出版式。
- 输出格式。

输出版式：

- 紧凑比赛版：一个模板接一个模板，强调节省页数，PDF 必须有页码。
- 书籍章节版：按章节展开，标题层级清楚，可读性更强。

输出格式：

- 只生成 Markdown。
- 只生成 PDF。
- 同时生成 Markdown 和 PDF。

## 12. PDF 页面控制

第一版 PDF 页面控制要达到一个硬标准：

目录中显示的页码必须能准确跳到对应标题所在页面。

具体要求：

- PDF 必须有页码。
- 目录页码必须和实际标题页一致。
- 标题不能被切到页面底部后正文跑到下一页，至少要避免明显断裂。
- 代码块必须可读，不能横向截断到不可用。
- 中文字体与等宽代码字体要稳定显示。

第一版不追求极限排版，但目录页码必须可信。

## 13. 离线要求

离线能力是必须项。

首次成功加载后，用户断网仍应能：

- 打开应用。
- 浏览已缓存的公共模板。
- 导入个人模板库 JSON。
- 新增、编辑、删除个人模板。
- 选择模板。
- 生成 Markdown。
- 生成 PDF。

应用应按 PWA/静态离线应用设计。

## 14. 技术方向

前端技术栈：

- Vue 3。
- TypeScript。
- Vite。
- shadcn-vue。
- Tailwind CSS。

Vue 编写风格：

- 使用 Composition API。
- 使用 `<script setup lang="ts">`。
- 状态逻辑优先拆到 composables。
- 页面组件负责组合，具体工作台区域拆成小组件。

拖拽排序方向：

- 优先评估 `vue-draggable-plus`。
- 简单单列表排序也可评估 VueUse 的 `useSortable`。
- 不管使用哪个拖拽库，都必须额外提供键盘移动按钮。

PDF 方向：

- Markdown 先作为稳定中间产物。
- PDF 从渲染后的文档生成。
- PDF 生成方案需要在原型阶段重点验证目录页码、页码、代码换页和中文字体。

## 15. 界面形态

第一屏是工作台：

- 左侧：公共模板库、个人模板库、分类树、搜索/筛选。
- 中间：当前打印稿大纲、已选条目、排序控制。
- 右侧：标题、目录、版式、导出格式、预览和下载。

个人模板编辑可以是弹窗，也可以是右侧或底部编辑面板。

整体设计应偏工具型：信息密度高、层级清楚、反复使用顺手。不要做成宣传页面。

## 16. 开源协议

这里先用人话解释：

- MIT 这类宽松协议：别人可以拿去用、改、商用，只要保留许可证说明。约束少。
- GPL-3.0 这类强 copyleft 协议：别人可以用、改、分发，但如果分发修改版或包含 GPL 内容的衍生作品，通常也要按 GPL-3.0 开源，并保留源码和许可证说明。

项目已决定采用 GPL-3.0。根目录需要保留标准 `LICENSE` 文件。

我们的参照仓库 `hh2048/XCPC` 是 GPL-3.0。如果后续直接复制、改写并分发其中内容，需要：

- 每个模板条目保留来源链接。
- `meta.json` 里记录来源和许可证。
- 不把来源不清楚的代码当成原创。

第一阶段整个项目使用 GPL-3.0，降低内容合规复杂度。等项目成熟后，如果特别想让应用代码使用 MIT，可以再把应用代码和内容库拆成两个仓库或两个包重新设计。

这不是法律意见，但作为开源项目启动策略足够稳。

## 17. 原型范围

第一版原型包含：

- 5 到 10 个原创或明确可用的样例模板。
- `板子/` 目录结构。
- 自动读取/生成公共模板索引的方案。
- 分类树或分组列表。
- 模板选择。
- 介绍详细度选择。
- 打印稿标题。
- 目录开关和目录深度选择。
- 学习顺序、字典序、手动排序。
- 鼠标拖拽排序。
- 键盘移动排序。
- 个人模板编辑器。
- 个人模板分类/文件夹。
- 个人模板 JSON 导入导出。
- Markdown 预览和下载。
- PDF 生成，且目录页码准确。
- 基础离线支持。

第一版暂不做大规模内容迁移。

## 18. 验收标准

原型完成时应满足：

- 用户不用登录也能完成选择、编辑、排序、导出。
- 公共模板库与应用在同一个仓库中，并按 `板子/分类/算法/meta.json + code.md` 组织。
- 用户能创建带分类路径的个人模板。
- 用户能导出并重新导入个人模板库 JSON。
- 用户断网后仍能打开应用并生成打印稿。
- Markdown 输出结构清晰，方便人工继续编辑。
- PDF 有页码。
- PDF 目录页码与实际标题页一致。
- 代码块可读，没有明显截断。
- 支持紧凑比赛版和书籍章节版。
- 支持拖拽排序，也支持键盘移动排序。

## 19. 已决策事项

- 公共模板库与应用放在同一个仓库。
- 公共模板库目录命名为 `板子/`。
- 每个算法模板底层目录包含 `meta.json` 和 `code.md`。
- 项目采用 GPL-3.0。
- PDF 原型必须保证目录页码准确。
- 个人模板第一版立即支持文件夹/分类。
- 手动排序同时支持拖拽和键盘移动。
- 项目使用 Vue 3 + shadcn-vue。

## 20. 待验证事项

- PDF 生成到底使用浏览器打印、Paged.js、Playwright 服务端方案，还是其他方案。
- `vue-draggable-plus` 与 shadcn-vue 组件组合时的体验。
- 离线状态下 PDF 生成需要缓存哪些字体和资源。
- GPL-3.0 下如何组织模板来源声明最清楚。
